## 问题描述

用户在使用AI分析功能时遇到400错误:

```
Client error '400 Bad Request' for url 'https://ark.cn-beijing.volces.com/api/v3/responses'
```

## 问题根源

通过代码分析发现:

1. **缺少豆包SDK依赖**

- `requirements.txt` 中没有 `volcenginesdkarkruntime` 包
- 代码尝试导入但失败,导致 `DOUBAO_SDK_AVAILABLE = False`
- 因此强制使用HTTP方式调用

2. **HTTP调用格式问题**

- 当前使用豆包原生API格式,但可能不符合最新的API要求
- 缺少错误日志详细输出,无法定位具体原因

3. **代码健壮性不足**

- SDK调用分支缺少try-except
- 无法优雅降级或提供详细错误信息

## 修复目标

确保AI分析功能能够正常调用豆包多模态API,支持两种调用方式:

- 方式1: 通过豆包SDK调用(推荐)
- 方式2: 通过HTTP直接调用(备用)

## 技术栈

**现有技术**:

- 后端: FastAPI + Python 3.x
- HTTP客户端: httpx
- 配置管理: pydantic-settings
- 日志: 自定义logger_service

**需要添加**:

- `volcenginesdkarkruntime` - 豆包官方SDK

## 实现方案

### 方案一: 安装SDK并修复调用(推荐)

**优点**:

- 使用官方SDK,更稳定可靠
- 自动处理认证和请求格式
- 支持更多高级功能

**实施步骤**:

1. 在 `requirements.txt` 添加 `volcenginesdkarkruntime`
2. 改进错误处理和降级逻辑
3. 添加详细日志输出

### 方案二: 修复HTTP调用格式

**优点**:

- 不依赖额外SDK
- 更灵活可控

**实施步骤**:

1. 研究豆包API文档,修正请求格式
2. 添加详细的错误响应日志
3. 测试验证

## 推荐方案: 混合策略

结合两种方案的优点:

1. 优先使用SDK(如果可用)
2. SDK失败时降级到HTTP调用
3. 添加完善的错误处理和日志
4. 两种方式都进行修复和测试

## 实现细节

### 1. requirements.txt 添加依赖

```python
# 豆包SDK
volcenginesdkarkruntime>=1.0.0
```

### 2. ai_service.py 改进

**当前问题**:

- SDK导入失败但没有明确提示
- SDK调用缺少异常处理
- HTTP调用格式可能不正确

**改进要点**:

- 添加SDK安装状态检查和日志
- SDK调用增加try-except
- 改进错误日志输出
- 添加请求/响应调试日志

### 3. 错误处理增强

**需要捕获的异常**:

- SDK导入失败
- SDK调用超时/网络错误
- API返回400/401/403等错误
- 响应格式解析失败

**日志增强**:

- 记录完整的请求URL
- 记录请求体(脱敏API Key)
- 记录完整的响应内容
- 记录错误详情
