# SmartAlbum 零停机部署指南

> **版本**: v1.0  
> **制定日期**: 2026-04-15  
> **适用环境**: Ubuntu 24.04 LTS + Docker Compose  
> **目标**: 实现100%可用性的无缝版本升级

---

## 一、部署策略概述

### 1.1 可选策略对比

| 策略 | 适用场景 | 复杂度 | 资源需求 | 停机时间 |
|------|----------|--------|----------|----------|
| **蓝绿部署** | 重大版本升级、数据库结构变更 | 中 | 2x资源 | 0 |
| **滚动更新** | 小版本更新、配置调整 | 低 | 1.5x资源 | 接近0 |
| **金丝雀发布** | 灰度测试、A/B测试 | 高 | 1.2x资源 | 0 |
| **重建部署** | 开发环境、测试环境 | 低 | 1x资源 | 5-30分钟 |

### 1.2 推荐策略：蓝绿部署

本方案采用**蓝绿部署**策略，原因如下：
- ✅ 可在任何时刻快速回滚
- ✅ 支持完整的功能验证后再切换流量
- ✅ 适合数据库结构变更的场景
- ✅ 部署过程对用户完全透明

```
┌─────────────────────────────────────────────────────────────────┐
│                        蓝绿部署流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   阶段1: 准备                    阶段2: 部署绿环境               │
│   ┌──────────┐                  ┌──────────────┐                │
│   │ 蓝环境运行 │                  │ 绿环境部署新 │                │
│   │  用户流量  │                  │   版本并测试  │                │
│   └────┬─────┘                  └──────┬───────┘                │
│        │                                │                        │
│        │ 流量100%                       │ 流量0%                 │

│                                     └──────────┘               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 二、架构设计

### 2.1 网络架构

```
                            用户请求
                               │
                               ▼
                    ┌─────────────────────┐
                    │   Nginx 负载均衡器    │
                    │   (端口: 80/443)     │
                    │   upstream 动态配置   │
                    └──────────┬──────────┘
                               │
            ┌──────────────────┼──────────────────┐
            │                  │                  │
            ▼                  ▼                  ▼
    ┌───────────────┐  ┌───────────────┐  ┌───────────────┐
    │   蓝环境      │  │   绿环境      │  │  共享服务      │
    │  (当前生产)   │  │   (新版本)    │  │               │
    ├───────────────┤  ├───────────────┤  ├───────────────┤
    │ Frontend:8081│  │ Frontend:8082 │  │  Redis:6379   │
    │ Backend:9999 │  │ Backend:9998  │  │  数据库:./data │
    │ Health:/api/ │  │ Health:/api/  │  │  存储:./storage│
    └───────────────┘  └───────────────┘  └───────────────┘
```

### 2.2 共享组件

以下组件在蓝绿环境间共享：
- **Redis**: 缓存、任务队列、会话存储
- **SQLite 数据库**: 数据持久化
- **存储目录**: 图片、缩略图
- **配置文件**: 环境变量

## 三、前置准备

### 3.1 系统要求

| 组件 | 蓝环境 | 绿环境 | 共享服务 | 总计 |
|------|--------|--------|----------|------|
| CPU | 1核 | 1核 | 0.5核 | 2.5核+ |
| 内存 | 1.5GB | 1.5GB | 512MB | 3.5GB+ |
| 磁盘 | 20GB | 20GB | 数据量x2 | 40GB+ |

### 3.2 端口规划

| 服务 | 蓝环境端口 | 绿环境端口 | 对外端口 |
|------|-----------|-----------|---------|
| Frontend | 8081 | 8082 | 80/443 |
| Backend | 9999 | 9998 | - |
| Redis | - | - | 6379 |

### 3.3 环境变量配置

创建 `.env.deployment` 文件：

```bash
# 部署配置
DEPLOYMENT_ENV=production
BLUE_FRONTEND_PORT=8081
BLUE_BACKEND_PORT=9999
GREEN_FRONTEND_PORT=8082
GREEN_BACKEND_PORT=9998
SHARED_REDIS_PORT=6379

# 健康检查配置
HEALTH_CHECK_TIMEOUT=60
HEALTH_CHECK_INTERVAL=5
MAX_RETRY_ATTEMPTS=12

# 流量切换配置
TRAFFIC_SWITCH_BATCH=10
TRAFFIC_SWITCH_INTERVAL=30
ROLLBACK_THRESHOLD_ERROR_RATE=1
ROLLBACK_THRESHOLD_LATENCY=5000
```

## 五、回滚策略

### 5.1 自动回滚触发条件

```yaml
自动回滚:
  触发条件:
    - 健康检查失败超过3次
    - 错误率 > 1% (持续2分钟)
    - P99延迟 > 5秒 (持续2分钟)
    - 5xx错误数量 > 10/分钟
    - 容器重启次数 > 3
  
  执行动作:
    - 立即停止流量切换
    - 回切流量到蓝环境
    - 通知运维团队
    - 保留绿环境用于诊断
```

## 八、故障排查

### 8.1 常见问题

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| 绿环境启动失败 | 配置错误、依赖缺失 | 查看日志，修复配置 |
| 健康检查失败 | 数据库连接问题 | 检查网络、数据库状态 |
| 流量切换后503 | 后端服务未就绪 | 增加健康检查等待时间 |
| 数据不一致 | 迁移脚本问题 | 回滚并修复迁移脚本 |
| 性能下降 | 资源不足 | 扩容或优化配置 |

### 8.2 日志位置

```
/var/log/smartalbum/
├── deploy/
│   ├── deploy-$(date).log          # 部署日志
│   ├── health-check-$(date).log    # 健康检查日志
│   └── rollback-$(date).log        # 回滚日志
├── backend/

└── nginx/

    └── error.log                   # Nginx 错误日志
```

## 九、最佳实践

1. **始终先执行备份**: 即使是最小的变更
2. **在测试环境验证**: 生产部署前必须完成
3. **保持团队沟通**: 部署期间保持实时通讯
4. **记录变更日志**: 每次部署都要记录
5. **定期检查备份**: 确保备份可恢复
6. **监控是关键**: 部署后至少观察24小时
